## Folder structure

```
src/
  index.ts              bootstrap: build app, listen, graceful shutdown (SIGTERM/SIGINT)
  app.ts                buildApp(): Fastify instance, registers plugins, error hook, routes
  features/
    health/
      routes.ts         GET /health
      index.ts          barrel export
    countries/          DB-backed country metadata catalog
      registry.ts       SEED data (USA/AUS/IDN) + pure helpers: canonicalizeCode, hashCountryFields, isSafePattern (ReDoS guard)
      repository.ts     ONLY db importer ‚Äî select/insert/update on countries table
      service.ts        async list/get metadata; buildAddressValidator() derives Zod from stored row; createCountry/updateCountry
      schemas.ts        metadata response + write-request Zod schemas
      seed.ts           idempotent seedCountries() (insert SEED_COUNTRIES, ON CONFLICT DO NOTHING)
      seed-cli.ts       `pnpm db:seed` entry ‚Äî seeds then closes the pool
      routes.ts         GET + POST /api/v1/countries, GET /:code/fields, PUT /:code
      index.ts          barrel (validator + metadata + write fns + canonicalizeCode)
    addresses/          capture + retrieve addresses
      schemas.ts        create request / address response Zod schemas
      repository.ts     ONLY db importer ‚Äî insert/list/findById on addresses table
```

## Layering rules

- `routes.ts` calls `service.ts`. Never the repository directly.
- `service.ts` calls `repository.ts`. Never the DB client directly.
- `repository.ts` is the **only** file that imports `db` from `shared/db/client.ts`.
- Features may import from `shared/`. Features **must not** import from another feature's internals ‚Äî only via that feature's `index.ts` barrel.
- `shared/` may not import from `features/`.
- Absolute imports use the `@/` alias (resolves to `src/`).

## Validation

Every external input goes through a Zod schema. **No exceptions.**

| Input source | Where the schema lives | Where parsing happens |
|---|---|---|
| HTTP request body / query / params | `features/<x>/schemas.ts` | route handler (via `fastify-type-provider-zod`) |
| Environment variables | `src/shared/config/env.ts` | once at boot |
| External API responses (e.g. Google Places) | `features/<x>/schemas.ts` | inside the fetcher in `service.ts` |

Reading `process.env.*` outside `src/shared/config/env.ts` is **forbidden**. ESLint `no-restricted-properties` rule enforces this.

## Logging

- **pino** only. **`console.log` is forbidden** in committed code (`no-console` ESLint rule).
- Import logger from `src/shared/logger.ts`; do not construct ad-hoc instances.
- Redacted keys: `authorization` header, `cookie` header, `password`, `token`, `secret`, `apiKey`.
- Development: pino-pretty (colorized). Production: JSON.
- Every log line carries `req.id` (Fastify built-in `genReqId` ‚Üí `x-request-id` header).

## Error handling

Throw typed `AppError` subclasses. The error handler (`src/plugins/error-handler.ts`) serializes to RFC 7807 `application/problem+json`:

```json
{
  "type": "about:blank",
  "title": "BAD_REQUEST",
  "status": 400,
  "detail": "validation failed",
  "instance": "/api/v1/addresses",
  "code": "BAD_REQUEST"
}
```

| Class | Status | When |
|---|---|---|
| `BadRequestError` | 400 | invalid input (Zod errors auto-wrap) |
| `UnauthorizedError` | 401 | missing/invalid auth |
| `ForbiddenError` | 403 | authenticated but not allowed |
| `NotFoundError` | 404 | resource doesn't exist |
| `ConflictError` | 409 | unique constraint, version mismatch |
| `InternalError` | 500 | unhandled ‚Äî don't throw directly |

Never `throw new Error(...)` from a route or service. Wrap or rethrow as `AppError`.

## Testing

- **Vitest** only. Uses Fastify's built-in `app.inject()` ‚Äî no real socket, faster than supertest.
- One test file per source file: `service.ts` ‚Üí `service.test.ts`, co-located.
- DB tests use a real Postgres (test schema or testcontainers). **No DB mocks.**
- `tests/setup.ts` sets `NODE_ENV=test`, `PORT=0`, and a dummy `DATABASE_URL`.

Do not strip the extensions.

## Conventions

- File naming: `kebab-case.ts`.
- Barrel exports: `index.ts` only at feature roots and package roots ‚Äî not deep.
- Imports order: external ‚Üí workspace ‚Üí `@/` absolute ‚Üí relative `./`.
- `strict: true`, `noUncheckedIndexedAccess: true`. Never `any`. Use `unknown` + narrowing.
- async/await only. No `.then()` chains.
- IDs: UUIDv7 for primary keys (default; actual choice is `senior-data-engineer`'s call).

## Database

- **Wired**: Postgres + Drizzle (via `drizzle-orm/postgres-js` + `postgres` driver).
- Connection: `src/shared/db/client.ts` reads `DATABASE_URL` from env.
- Schema: `src/shared/db/schema.ts` ‚Äî **empty at scaffold**. Authoring is `senior-data-engineer`'s job.
- Migrations: `pnpm db:generate` after schema changes, `pnpm db:migrate` to apply.
- Seed: after `db:migrate`, run `pnpm db:seed` to load the built-in countries (idempotent).
- **Countries are DB-backed and runtime-editable** via `POST/PUT /api/v1/countries`. `registry.ts` holds only the SEED data + pure helpers; the live source of truth is the `countries` table. ‚öÝÔ∏è The write path is **intentionally unauthenticated for now** (handoff "Guard 2" deferred) ‚Äî add an admin guard (`senior-security-engineer`) before exposing it beyond a trusted network. ReDoS validation on submitted regex patterns (Guard 1, `isSafePattern`) **is** enforced and must stay.
- Local Postgres: `docker compose up -d db` (Port 5432, user/pass/db: `app`).
  - If a host Postgres already holds `localhost:5432`, it shadows the container and DB commands hit the wrong DB (`role "app" does not exist`). Use a gitignored `docker-compose.override.yml` mapping `5433:5432` + `DATABASE_URL=...@localhost:5433/app` in `.env`. See README "Local Postgres on port 5432".
  - `drizzle.config.ts` loads `.env` via `process.loadEnvFile` so `db:generate`/`db:migrate`/`db:studio` honor the same `DATABASE_URL` as the app (drizzle-kit does not auto-load `.env`).

## Adding a new feature

1. `mkdir -p src/features/<name>` ‚Üí `routes.ts`, `service.ts`, `repository.ts`, `schemas.ts`, `index.ts`
2. Define request/response Zod schemas in `schemas.ts`
3. Implement `repository.ts` against the Drizzle `db` client
4. Implement `service.ts` ‚Äî pure logic, no framework imports
5. Wire `routes.ts` ‚Äî parse with schemas, call service, return
6. Register routes in `app.ts` with `typed.register(<name>Routes)`
7. Write tests co-located with each source file
8. Verify in `/docs` (OpenAPI auto-generates from Zod schemas)

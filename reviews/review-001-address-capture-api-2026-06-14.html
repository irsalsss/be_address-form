MEDIUMReliability · Limits

GET /addresses returns the whole table — no pagination, no LIMIT

src/features/addresses/repository.ts:22 · src/features/addresses/routes.ts (GET /api/v1/addresses)

listAddresses() runs select().from(addresses).orderBy(desc(createdAt)) with no .limit(), so the route sends back every row in one big JSON array.

Why it matters: Fine with a few rows, but it's the classic "fetch everything" endpoint that falls over in production. Once the table has a lot of rows, one request loads the whole thing into memory, builds a multi-MB response, and blocks Node while it does — a single call can slow down the whole server. There's also no cap, so nothing stops someone hammering it. This jumps to High once the table can grow without limits and is publicly reachable.

Fix: Add keyset (cursor) pagination, or at minimum a hard cap. 
// repository.ts
export async function listAddresses(limit = 50, before?: Date) {
  const q = db.select().from(addresses)
    .orderBy(desc(addresses.createdAt), desc(addresses.id))
    .limit(Math.min(limit, 100));
  return before ? q.where(lt(addresses.createdAt, before)) : q;
}
// route: parse ?limit & ?before via a Zod query schema, return a nextCursor
MEDIUMReliability · DB

List order isn't stable when two rows share the same created_at

src/features/addresses/repository.ts:22

Sorting is by desc(createdAt) only. The primary key is defaultRandom() (uuid v4), which has no time built in, so two rows saved at the same instant have nothing to break the tie — their order can come out either way.

Why it matters: This already happened in this build — the repository test had two inserts land on the same now(), the order flipped, and a fake 5 ms sleep was added to make it pass. In production it means the same request can return rows in a different order each time. And once pagination (above) is added, a tie sitting right on a page edge can drop or duplicate a row in the results.

Fix: Add a second sort key to break ties, and/or switch to a time-ordered primary key (uuid v7), which CLAUDE.md already lists as the default. 
.orderBy(desc(addresses.createdAt), desc(addresses.id))
// + when on PG18 / app-side gen: id uuid PK default uuidv7() → ordering & tiebreak in one
MEDIUMSecurity · Validation

Free-text address fields have no maximum length

src/features/countries/service.ts:56 (required text → z.string().trim().min(1)) · src/features/addresses/schemas.ts:15 (fields: z.record(z.string(), z.string()))

The validator built from the registry limits postal codes and dropdowns, but every free-text field (line1, city, suburb, street, district, …) only checks min(1) with no max. Whatever the client sends gets saved as-is into the fields jsonb column.

Why it matters: A client can send a ~1 MB string in any text field (the only limit is Fastify's default 1 MB body cap) and it's stored as-is. Easy way to bloat rows and waste storage, and those huge values come right back out through the list endpoint — which makes the no-pagination issue worse. No money or code-execution risk, so Medium not High.

Fix: Add a max length in the registry-built schema, and lower the body size limit for this route. 
// service.ts fieldSchema — text branch
base = z.string().trim().min(1).max(field.validation?.maxLength ?? 200);
// optionally: app.register(addressesRoutes, { bodyLimit: 16 * 1024 })

LOWSecurity · Authz

No auth on the read or write endpoints

src/features/addresses/routes.ts (POST /api/v1/addresses, GET /api/v1/addresses[/:id])

There's no auth check on reads or writes. Anyone who can reach the service can add rows and read every stored address.

Why it matters: This is a documented demo choice in spec.md (open endpoints), so it's Low, not High — but it has to be fixed before any real deploy. Addresses are personal data (PII): an open GET list hands out everyone's data at once, and an open POST lets anyone write — which combined with the no-max-length issue makes it trivial to fill up storage. Flagging it so it's a deliberate call at PR time, not something that slipped through.

Fix: Before prod, put both routes behind auth (owned by senior-security-engineer per CLAUDE.md), limit reads to the logged-in user, and rate-limit POST.
LOWSecurity · Validation

Request body accepts any keys before the strict per-country check rejects them

src/features/addresses/schemas.ts:15

The Fastify body schema is fields: z.record(z.string(), z.string()), which accepts any number of arbitrary keys. The strict, country-specific check only runs later in the service, where unknown keys get rejected.

Why it matters: The end result is correct (junk gets a 400), but only after Fastify has parsed and Zod has walked a potentially big map — wasted work on a request that was always going to fail. The body size limit keeps it bounded, so it's Low. Worth noting because it's basically free to fix.

Fix: Cap how many keys are allowed (pair it with the per-field max above). 
fields: z.record(z.string().max(40), z.string().max(500))
  .refine((o) => Object.keys(o).length <= 12, "too many fields")
